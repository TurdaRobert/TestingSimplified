Bug

An error, flaw, or fault in a Software System that causes it to fail to perform as required.
Bugs are inevitable part of software development, and threaten the success of any software engineering endeavor.

Detected:An error, fault or flaw in the Software System is observed and logged.

Located:The cause of the Bug in the Software System has been found.

Fixed:The Bug has been removed from the Software System.

Closed:The removal of the Bug from the Software System has been confirmed.

The priority and impact of the bug is clearly described.
The bug clearly identifies the location of the problem and the circumstances under which it was discovered.
The bug can be readily reproduced from the information provided.
The impact that the bug has on subsequent development activities is understood.

The evaluation of the bugs aims at ensuring that they are actual bugs and not just misinterpretations of the systemâ€™s behavior or specification.

Test

A test:
Verifies an aspect of the system
Validates that the functionality of the system has been appropriately implemented
Demonstrates that the system provides the desired quality of service.

A test is not limited to verifying only functional requirements. Tests are defined to verify any aspect of the system, including performance, load, reliability, etc. Tests are often repeated to observe changes in results. See the Hints and Tips section for more information.

Defined:A test must be conceived and scoped to fit it into the projectâ€™s approach to testing. This allows the project to make the best use of its test resources and test cases by identifying appropriate testing of the software system.

Specified:A test must be specified prior to its execution. This describes how the test is supposed to be executed and evaluated.

Executed:A test is executed to enable the team to draw conclusions about the software systemâ€™s quality.

Evaluated:A test can either pass successfully, or fail, or yield an undetermined result. If a test passes, then the elements (requirements, software etc.) being verified can change state to indicate that they have been verified. If a test fails, then one or more bugs are created to describe the problems encountered. If the result of a test is undetermined, actions may be required to understand and fix what is causing this before the test can be evaluated.

The test addresses specific requirements.
The test produces a set of concrete results that can be analyzed and evaluated to determine the testâ€™s outcome.
The test can be repeated an arbitrary number of times producing the same results on the same execution infrastructure.

Evaluating the test includes evaluating both the definition of the test as well as the results of the execution of the test. A test must be valuable to the development effort and it must be adequate to verify the targeted requirements and optionally offer diagnostic information during execution.

Test Case

Test cases:
Provide the building blocks for designing and implementing tests.
Provide a mechanism to complete and verify the requirements.
Allow tests to be specified before implementation starts.
Provide a way to assess system quality.
The best test cases describe test scenarios, where a scenario is a complete thread of execution through the system.

Test Ideas Formulated:The lightest level of detail just captures the initial idea that will inform the test case. When defining a test case it needs to be clear what the idea behind the test case is, and which slice of the requirements it applies to. A test case can be considered as a question that you ask of the system. The point of running the test case is to gain information, for example whether the system works as specified or not. There has to be a reason for the creation of the test case. Some test cases exist to address specific project risks, whereas others are based upon experience, hunches, test results or test techniques used to methodically produce test cases. This reasoning can be summarized as the test idea behind the test case, and is the earliest form that a test case can take.

Scenario Chosen:To make the test case executable the scenario to be performed needs to be selected. Once the scenario has been identified the test case is defined enough to support exploratory and investigative testing. This can be very useful early in the project lifecycle when the insight provided by testing the system is invaluable but the specification (and solution) may not be stable enough to support formal, scripted testing.

Variables Set:Once the variables have been identified the test case can be further elaborated to explicitly define specific values for all of the variables involved in the test case.

Scripted or Automated:If a test case is to be used many times or to support many different tests, then it is worth making the effort to fully script or automate it. For a test case to be fully scripted and automatable it must contain no ambiguity. At this level of detail the test case is often documented as an automated testing script, one that can be interpreted and executed by an automated testing tool.

1..N Pre-conditions: Pre-conditions that specify the state of the system before the test.
1..N Input: The messages and data to be received at the specific inputs in the scenario. For the early levels of detail this may just be the initial data needed to start the scenario.
1 Test Scenario: The scenario that the test case will exercise.
1..N Observations (with Expected Results): The observations to be taken as the test case is performed including the expected result. This includes describing any desired system post-conditions, and how the expected results will be verified.

The pre-conditions must be sufficient to enable the test to be performed.
The test case defines a complete scenario.
The input matches the scenario.
The expected results are adequate and verifiable.
The test case reflects at least one requirement on the system.
The scenario and the underlying requirements to be tested are clearly defined.

Test Results

Test results:
Provide objective evidence of the quality of the system produced.
Allow defects to be identified.
Provide verification that the system behaves as required.

Generated:As tests are executed, they generate data that is collected for analysis and evaluation.

Analyzed:Test results are analyzed to determine whether the test was successful or not.

Evaluated:Individual tests are part of a broader effort to evaluate the software system under construction.

1 Test Instance Description: A description of the particulars of the test execution including the software build it was run against, the date and time it was executed and any other relevant information.
1..N Results of Tests: The results of the test, including the test data and a description of whether the test executed successfully or not. If the test failed, this provides a reference to the bugs created.
1 Test Evaluation: An evaluation of the test results in the context of the overall test objectives of the current work cycle.

The results of the test are clearly stated as passed, failed or undetermined.
The date and time that the test was executed are recorded.
The circumstances (test environment, build etc.) under which the test was executed are recorded.
The data generated by the test has been recorded in the appropriate format.

Test Specification

A test specification:
Defines the objectives and success criteria for a test.
Describes how to execute a set of test cases to verify a software system.

Objectives Defined:Each test has a specific set of test objectives that clearly define the goals of the test. If a test does not have a clear set of test objectives it is unlikely to be effective.

Approach Identified:There are many different approaches that can be adopted when testing a system. At one end of the scale you have exploratory testing where, given a clearly defined test objective, the tester can improvise and create the appropriate test cases on the fly. At the other end of the scale you have rigorously scripted tests where the tester has to follow the pre-scribed instructions to the letter. And there are many variations in between these two extremes.

Test Cases Identified:Generally the test objectives are achieved by executing one or more test cases. If the test is to be repeatable then the test cases need to be specifically identified.

Orchestrated:Tests can become very large and very complicated. For example, a complete system test can involve running all the previously defined test cases, or a business test may involve executing a large number of test cases in a particular sequence or with particular timing.

1..N Test Objectives: A clear statement of the objectives of the test. These objectives should refer to the specific requirements that are being addressed by the test and also explain to what degree those requirements need to be fulfilled to meet each objective.
1..N Evaluation Criteria: Details of how the test results will be evaluated and the test coverage required.
1 Test Environment Description: A description or illustration of the hardware and software environment in which to perform the test.
1..N Test Instructions: A description of the procedures to be followed to execute the set of test cases. This may take the form of an automated test suite when orchestrating a set of automated test cases.

The objective clearly states the purpose of the test.
The evaluation criteria unambiguously define the criteria for a successful test execution.
The test specification describes the test procedures and environment in enough detail that the test can be executed multiple times, producing predictable results.
The test specification describes the data that is collected as part of the test execution, including how it is collected and stored.

Identify Test

This activity results in a set of test specifications that define the tests to be run and the test approach to be adopted.
This activity is repeated many times during the lifecycle of a project. The most critical tests are identified early in the lifecycle to focus the team towards verifying the most important decisions made for the software system. Additional test objectives may need to be added to verify any changing requirements or improve the testing process.

Test Defined
At the conclusion of this activity, one or more tests have been specified and their objectives defined in the test specification. This includes progressing to the following alpha states:
Requirements to Acceptable (contributes to)
Software System to Architecture Selected (contributes to) and Demonstrable (contributes to) and Ready (contributes to) and Usable (contributes to)
Test to Defined

This also includes producing the following work product levels of detail:
Test Specification to Objectives Defined or beyond

This activity needs to be a group activity involving a number of complementary competencies to define a sensible set of tests for the project.
The identification of the tests will be led by someone with the Testing competency. Additional assistance will be required from people with the Leadership and Development competencies.

Testing
The individual accountable for this activity has to:
Shape the testing objectives and procedures.
Document the test specification.
Ensure coverage of the appropriate test cases.

Leadership
Test objectives are created to verify the achievement of a project milestone and must be achievable within the time and budget constraints of the project. A team member with Leadership competency must participate in test identification to ensure that the completion of the identified tests is possible within the project constraints.

Development
Development knowledge is also required to define the right test objectives based on the technical solution approach and identified areas of technical risk.

Build Test

Build a test to:
Align with the test objectives.
Create the necessary test procedures and instructions.
Define how the outcomes of the test are evaluated to determine the test results.
Automate its execution.

This activity results in a test specification that specifies a test based on one or more test cases.
This activity is repeated multiple times during the lifecycle of a project. Tests for the most critical scenarios and non-functional requirements are specified early in the lifecycle to focus the team on verifying the most important decisions made for the system. These test specifications will need to be kept up to date as the project progresses, bugs are found and the requirements change.

Test Specified

At the conclusion of this activity, the test specification has been developed to specify the test by identifying the test cases that the test incorporates. This includes progressing to the following alpha states:
Software System to Demonstrable (contributes to) and Usable (contributes to) and Ready (contributes to)
Test to Specified
Bug to Fixed (optional) and Closed (optional)

This also includes producing the following work product levels of detail:
Test Specification to Approach Identified or beyond

The activity will be led by someone with the Tester competency.
Additional assistance may be required from people with the Developer competency.

Testing
The individual accountable for this activity has to:
Document the test specification.
Ensure the selection of appropriate test cases.

Development
Where automated or programmatic testing methods are adopted, individuals will be needed to set up, maintain and develop the automated tests.

Execute Test

Execute a test to:
Verify that the software system meets the evaluation criteria of the test.
Identify bugs in the software system.

This activity results in the test being executed and evaluated, and its test results being generated and analyzed. If the test fails, one or more bugs are identified and handled by the project as appropriate.

Test Evaluated

At the conclusion of a test, the results of the test have been analyzed and evaluated. This includes progressing to the following alpha states:
Requirements to Addressed (contributes to) and Fulfilled (contributes to)
Software System to Demonstrable (contributes to) and Usable (contributes to) and Ready (contributes to) and Operational (contributes to)
Test to Executed and Evaluated
Bug to Detected (optional: Bugs have been identified as a result of test failure) and Fixed and Closed (optional)

This also includes producing the following work product levels of detail:
Test Results to Analyzed or beyond

Executing tests should not be seen as purely the domain of specialist test resources, everybody involved in the project should be able to achieve the Testing competency and contribute to the testing of the system.

Testing
The individual accountable for this activity has to:
Execute the test.
Gather and store the test outputs.
Provide an objective evaluation of the test results.
Raise bugs as appropriate.
Operate specialized testing software and/or equipment.

Development
Often a certain level of Development competency will be required to support the analysis of the test results and the correct identification of defects.
The level of developer competency required is dependent on the type of test being executed, the complexity of the system under test and the type of defect discovered.
